Operating rhythm

Observe, route, act, verify, project.

LUX deliberately slows automation down enough to make each claim inspectable.

01

Observe

Read `.lux/`, engine status, logs, hierarchy, screenshots, and recent run evidence.

02

Route

Choose the verified engine surface instead of assuming every adapter is equally mature.

03

Act

Run commands through gateway-owned CLI, HTTP/WebSocket, MCP, or bridge paths.

04

Verify

Capture compile, test, run, status, screenshot, or log evidence before completion.

05

Project

Publish only proven state into docs, skills, CLI summaries, and `.lux/` projections.

Repository anatomy

Each layer owns one kind of truth.

.lux/

Canonical runtime truth: specs, tickets, events, roadmap, sessions, and run evidence.

gateway/

Rust CLI, Axum HTTP/WS server, MCP tools, routing, and engine orchestration.

bridge/

Ordinary in-repository engine bridge source. No git submodule or external bridge remote.

Skills/

Manifest-backed workflow library projected into target projects with capability guardrails.

docs/

Human-readable projection of runtime and product reality, never the SSoT over `.lux/`.

scripts/

Local verification, policy scanning, structure checks, and release maintenance.

Capability tiers

Unity is verified. Everything else says exactly what it can prove.

verified

Unity

Primary public-beta path for bridge install, status, compile, tests, run evidence, and screenshots.

partial

Godot

Detection, bridge install, status, and workflow projection only; build/run/test remain unsupported.

planned

Three.js

Adapter files may exist, but runtime automation remains planned until a harness is present and verified.
